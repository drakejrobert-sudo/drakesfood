# Recipe Submission API Contract

This contract defines the V1 recipe submission API for `drakesfood.com`. V1 accepts text and links only. Submissions are stored for Drake to review manually and must never auto-publish to the website.

See `recipe-submission-system.md` for the end-to-end system documentation, including frontend wiring, infrastructure, deployment, testing, and logs.

## Endpoint

```http
POST /recipe-submissions
Content-Type: application/json
```

The production API base URL will be provided by the OpenTofu-managed API Gateway output and passed to the Angular app configuration when live wiring is added.

## Request Body

```json
{
  "title": "Smoked birria tacos",
  "description": "Would love to see this cooked on the Masterbuilt Gravity Series 800.",
  "name": "Jane",
  "email": "jane@example.com",
  "recipeUrl": "https://example.com/recipe",
  "socialUrl": "https://instagram.com/example",
  "permissionToCredit": true,
  "website": ""
}
```

## Fields

| Field | Required | Type | Max length | Notes |
| --- | --- | --- | --- | --- |
| `title` | Yes | string | 120 | Recipe or idea title. Trim before validation and storage. |
| `description` | Yes | string | 3000 | Recipe notes, challenge, context, or cooking idea. Trim before validation and storage. |
| `name` | No | string | 100 | Submitter name or handle. Trim before storage. |
| `email` | No | string | 254 | Optional follow-up email. Reject clearly invalid email values when provided. |
| `recipeUrl` | No | string | 500 | Optional source recipe, video, or notes URL. Reject clearly invalid URL values when provided. |
| `socialUrl` | No | string | 500 | Optional social post or creator URL. Reject clearly invalid URL values when provided. |
| `permissionToCredit` | No | boolean | n/a | Defaults to `false` when omitted. |
| `website` | Yes | string | n/a | Hidden honeypot field. Normal browser submissions should send an empty string. |

Clients should send all fields shown above. The backend should tolerate omitted optional fields and normalize them to empty values or omitted DynamoDB attributes consistently.

## Success Response

```http
HTTP/1.1 200 OK
Content-Type: application/json
```

```json
{
  "success": true,
  "message": "Thanks! Your idea was sent to Drake."
}
```

## Validation Error Response

```http
HTTP/1.1 400 Bad Request
Content-Type: application/json
```

```json
{
  "success": false,
  "message": "Please include a recipe title and description."
}
```

Use this response for missing required fields, empty required fields after trimming, invalid field types, invalid optional email or URL values, and field length violations. User-facing messages should stay friendly and should not expose internal implementation details.

## Status Codes

| Status | Use |
| --- | --- |
| `200 OK` | Valid submission accepted and stored. Also use for populated honeypot submissions so bots do not receive useful feedback. |
| `400 Bad Request` | Malformed JSON, invalid field types, missing required fields, invalid optional email or URL values, or field length violations. |
| `405 Method Not Allowed` | Any method other than `POST`, unless API Gateway handles unsupported routes before Lambda. |
| `413 Payload Too Large` | Request body exceeds the configured API Gateway or Lambda body size limit, when this can be configured separately. |
| `415 Unsupported Media Type` | Request does not use `application/json`, if content-type enforcement is implemented. |
| `500 Internal Server Error` | Unexpected server failure. Return a generic message with no stack trace or AWS details. |

## Honeypot Behavior

`website` is a hidden spam-trap field. If `website` contains a non-empty value after trimming, the backend should:

- Return the same generic success response used for normal successful submissions.
- Not create a normal DynamoDB submission item.
- Not send email notifications.
- Not reveal that the request was classified as spam.

Server-side honeypot enforcement is required even though the Angular form also includes frontend handling.

## DynamoDB Item Shape

Successful non-honeypot submissions should be stored as one item keyed by `submissionId`.

```json
{
  "submissionId": "uuid",
  "submittedAt": "2026-04-30T14:30:00.000Z",
  "status": "new",
  "title": "Smoked birria tacos",
  "description": "Would love to see this cooked on the Masterbuilt Gravity Series 800.",
  "name": "Jane",
  "email": "jane@example.com",
  "recipeUrl": "https://example.com/recipe",
  "socialUrl": "https://instagram.com/example",
  "permissionToCredit": true,
  "source": "drakesfood.com"
}
```

Required item attributes:

- `submissionId`: UUID generated by the Lambda handler.
- `submittedAt`: ISO 8601 timestamp generated by the Lambda handler.
- `status`: Always `new` for V1 accepted submissions.
- `title`: Trimmed title.
- `description`: Trimmed description.
- `permissionToCredit`: Boolean value, defaulting to `false`.
- `source`: `drakesfood.com` unless overridden by environment configuration for local testing.

Optional item attributes:

- `name`
- `email`
- `recipeUrl`
- `socialUrl`

Optional string attributes may be omitted or stored as empty strings, but the Lambda handler should use one consistent approach.

## Security And Privacy Notes

- Validate every field server-side; frontend validation is only a usability aid.
- Trim string fields before validation and storage.
- Do not store the `website` honeypot field on accepted submission items.
- Do not return stack traces, AWS request IDs, table names, function names, or raw SDK errors to users.
- Restrict browser CORS in infrastructure to approved production and local development origins.
- Keep CAPTCHA out of V1 unless real abuse requires it later.
- Do not add file or image uploads in V1.
